Part A

The Merging Problem

Forget sorting for a moment. Solve this smaller problem first.

Part A · The Problem

Two Sorted Arrays, One Sorted Result

You are given two arrays that are already sorted. Produce a single sorted array containing all their values.

Input

A = 1 4 7 9 B = 2 3 8

Output

C = 1 2 3 4 7 8 9

The important word is already. Because each array is sorted, the smallest value that has not been used yet must be at the front of one of them.

Part A · The Idea

Two Pointers, Take the Smaller

The method

  • Put a pointer i at the start of A, and j at the start of B.
  • Compare A[i] with B[j].
  • Copy the smaller one into C.
  • Advance only that pointer.
  • Repeat until one array runs out.

Why it works

Each array is sorted, so A[i] is the smallest unused value of A, and B[j] the smallest of B. The smaller of those two must be the smallest value remaining overall.

Cost: every value is copied exactly once, so merging takes about nA + nB steps.

Part A · The Picture

The Two Pointers in Action

After copying 1 and 2: we compare A[1] = 4 with B[1] = 3, and take the 3.

Part A · The Code

The Main Loop, and the Leftovers

int i = 0, j = 0, k = 0;

// while BOTH arrays still have values
while (i < nA && j < nB) {
    if (A[i] <= B[j])
        C[k++] = A[i++];   // take from A
    else
        C[k++] = B[j++];   // take from B
}

// one array is empty — copy whatever is left
while (i < nA) C[k++] = A[i++];
while (j < nB) C[k++] = B[j++];

Why two extra loops?

The main loop stops as soon as one array runs out. The other may still hold values — and they are already sorted, so we just copy them across.

Only one of the two leftover loops ever runs.

Using <= rather than < keeps equal values in their original order.

Worked Example 1

Merging Two Sorted Arrays

#include <iostream>
using namespace std;

int main() {
    int A[4] = {1, 4, 7, 9};
    int B[3] = {2, 3, 8};
    int nA = 4, nB = 3;
    int C[7];

    int i = 0, j = 0, k = 0;
    while (i < nA && j < nB) {
        if (A[i] <= B[j]) C[k++] = A[i++];
        else              C[k++] = B[j++];
    }
    while (i < nA) C[k++] = A[i++];   // leftovers
    while (j < nB) C[k++] = B[j++];

    for (int x = 0; x < 7; x++) cout << C[x] << " ";
    return 0;
}
Expected Output
1 2 3 4 7 8 9
What happened
1 vs 2 → take 1 4 vs 2 → take 2 4 vs 3 → take 3 4 vs 8 → take 4 7 vs 8 → take 7 9 vs 8 → take 8 B is empty → copy 9

6 comparisons, then one leftover.

Part B

From Merging to Merge Sort

Now use that solution to sort a whole array.

Part B · The Insight

If the Halves Were Sorted, We Could Merge Them

We can merge two sorted arrays quickly. So imagine cutting an unsorted array in half…

unsorted: 5 2 8 1 left half: 5 2 right half: 8 1 ↓ somehow sort ↓ somehow sort 2 5 1 8 merge them → 1 2 5 8

But how do we sort each half? The same way — cut it in half, sort those halves, and merge.

That is recursion. And a half of a half of a half eventually has just one value, which is already sorted.

Part B · The Strategy

Divide and Conquer, in Three Steps

The algorithm

  • Divide: cut the array in half at mid.
  • Conquer: call mergeSort on each half.
  • Combine: merge the two sorted halves.

The base case

An array of one value — or none — is already sorted. Stop and return.

if (left >= right) return;

Notice: the splitting does no sorting at all. All the sorting happens on the way back up, inside merge.

Part B · The Picture

Split Going Down, Merge Coming Up

Part B · The Code

The mergeSort Function

void mergeSort(int arr[], int left, int right) {
    // base case: 0 or 1 value is already sorted
    if (left >= right) return;

    int mid = left + (right - left) / 2;

    mergeSort(arr, left, mid);        // sort left half
    mergeSort(arr, mid + 1, right);   // sort right half

    merge(arr, left, mid, right);     // combine them
}
  • left and right mark the piece we are sorting.
  • mid splits it: left..mid and mid+1..right.
  • Both calls must finish before merge runs.

Only four lines — because all the real work lives in merge, which we already wrote in Part A.

Call it with mergeSort(arr, 0, n - 1);

Part B · The Code

Merging Two Halves of One Array

void merge(int arr[], int left, int mid, int right) {
    int n1 = mid - left + 1;      // size of left half
    int n2 = right - mid;         // size of right half

    int L[100], R[100];           // temporary copies
    for (int i = 0; i < n1; i++) L[i] = arr[left + i];
    for (int j = 0; j < n2; j++) R[j] = arr[mid + 1 + j];

    int i = 0, j = 0, k = left;   // k writes back into arr

    while (i < n1 && j < n2) {
        if (L[i] <= R[j]) arr[k++] = L[i++];
        else              arr[k++] = R[j++];
    }
    while (i < n1) arr[k++] = L[i++];
    while (j < n2) arr[k++] = R[j++];
}

This is Part A's merge, with two changes:

  • The two halves live inside one array, so we first copy them out into L and R.
  • The result is written back into arr, starting at index left.

The copies are needed: writing into arr would otherwise destroy values we still have to read.

Worked Example 2

The Complete Program

#include <iostream>
using namespace std;

void merge(int arr[], int left, int mid, int right) {
    int n1 = mid - left + 1, n2 = right - mid;
    int L[100], R[100];
    for (int i = 0; i < n1; i++) L[i] = arr[left + i];
    for (int j = 0; j < n2; j++) R[j] = arr[mid + 1 + j];

    int i = 0, j = 0, k = left;
    while (i < n1 && j < n2) {
        if (L[i] <= R[j]) arr[k++] = L[i++];
        else              arr[k++] = R[j++];
    }
    while (i < n1) arr[k++] = L[i++];
    while (j < n2) arr[k++] = R[j++];
}

void mergeSort(int arr[], int left, int right) {
    if (left >= right) return;
    int mid = left + (right - left) / 2;
    mergeSort(arr, left, mid);
    mergeSort(arr, mid + 1, right);
    merge(arr, left, mid, right);
}

int main() {
    int arr[7] = {38, 27, 43, 3, 9, 82, 10};
    mergeSort(arr, 0, 6);
    for (int i = 0; i < 7; i++) cout << arr[i] << " ";
    return 0;
}
Expected Output
3 9 10 27 38 43 82

What it does: mergeSort keeps splitting until every piece holds one value, then merge rebuilds the array in order.

Two short functions sort any array of any size.

Worked Example 3

Tracing the Calls

Splits and merges
split [38 27 43 3 9 82 10] split [38 27 43 3] split [38 27] merged -> [27 38] split [43 3] merged -> [3 43] merged -> [3 27 38 43] split [9 82 10] split [9 82] merged -> [9 82] merged -> [9 10 82] merged -> [3 9 10 27 38 43 82]

What to notice

  • Every split happens before any merge in that branch.
  • The left half is fully sorted before the right half is even touched.
  • The last merge joins two sorted halves of four and three values.

Odd sizes are fine: [9 82 10] splits into [9 82] and [10]. A single value needs no sorting.

Part B · The Cost

Why Merge Sort Is Fast

Each level of the tree merges every value exactly once — that is n work. The number of levels is how many times you can halve n.

Putting It Together

Compared With the Simple Sorts

For one million values

bubble sort : about 1,000,000,000,000 steps merge sort : about 20,000,000 steps

Merge sort trades a little extra memory for an enormous gain in speed.

Be Careful

Common Mistakes

  • Forgetting the leftover loops
    the tail of the longer array is silently dropped
  • Advancing both pointers
    only the pointer you copied from moves forward
  • No base case
    if (left >= right) return; must come first
  • Wrong halves
    use left..mid and mid+1..right, never mid..right
  • Merging before both calls return
    merge comes after the two recursive calls
  • Writing into arr while still reading it
    copy the halves into L and R first

The infinite-recursion test: with mid..right, a two-element piece never shrinks — the program crashes.

5 · Why copy first?

Because merge writes its output back into arr starting at left. Without the copies, the very first write would overwrite a value in the left half that has not been read yet.

6 · Levels for a million

About 20. Each level halves the size, and 2²⁰ is just over one million, so twenty halvings reach pieces of size one.

Wrap Up

Summary

  • Merging joins two sorted arrays with two pointers: compare the fronts, take the smaller, advance that pointer.
  • After the main loop, copy the leftovers from whichever array still has values.
  • Merge sort = divide at mid, sort each half by recursion, then merge.
  • The base case is a piece of one value, which is already sorted.
  • Splitting does no sorting — all the sorting happens inside merge, on the way back up.
  • Cost: log₂(n) levels × n work = n × log₂(n), far faster than n × n.

Solve the small problem first — then sorting is just merging, applied again and again.
