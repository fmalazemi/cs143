Recall: an object holds attributes (data) and methods (behaviour). Encapsulation decides who is allowed to touch them.

The Idea

What Is Encapsulation?

Encapsulation means wrapping the data and the methods together, and hiding the data inside. Outside code uses the object only through its public methods.

The Three Keywords

Access Levels

KeywordWho can use itTypical use
publicany code, inside or outside the classthe methods others call (the interface)
privateonly the class's own methodsthe data you want to protect
protectedthe class and classes built from itfor inheritance (covered later)

Common pattern: make the data private and the methods public. This is the heart of encapsulation.

The Syntax

Marking Members Private and Public

class BankAccount {
private:
    double balance = 0;     // hidden data

public:
    void deposit(double amount) {
        balance = balance + amount;
    }
    double getBalance() {
        return balance;
    }
};
  • Everything under private: is hidden from outside.
  • Everything under public: can be used from outside.
  • In a class, members are private by default until you write public:.

We set balance = 0 here to give it a starting value; constructors (a cleaner way) come later.

The Motivation

Why Hide the Data?

If data were public…

account.balance = -9999;  // anyone can break it

Outside code could set a meaningless or illegal value, leaving the object in a broken state.

With data private…

account.deposit(50);      // only safe paths
// account.balance = -9999;  // ERROR: private

Changes must go through methods that can check the value first.

Encapsulation protects an object by controlling exactly how its data can change.

Who Can Reach What

Allowed vs. Blocked

From outside, code may call the public methods, but it cannot touch the private data directly.

The Safe Doorways

Getters and Setters

class Student {
private:
    int age = 0;

public:
    int getAge() {          // getter: read the data
        return age;
    }
    void setAge(int a) {    // setter: change the data
        age = a;
    }
};
  • A getter lets outside code read a private value.
  • A setter lets outside code change it — through a controlled door.
  • The class decides what is allowed; the data stays private.

Next: the real power of a setter is that it can check the value first.

The Real Payoff

Validation in a Setter

class Student {
private:
    int age = 0;
public:
    void setAge(int a) {
        if (a >= 0 && a <= 120)   // check first
            age = a;              // only set if valid
        // otherwise ignore the bad value
    }
    int getAge() { return age; }
};
Effect
setAge(20) -> age is 20 setAge(-5) -> ignored, age stays

What it does: the setter rejects an impossible age. The object can never hold an invalid value.

This is the point of encapsulation: the class guards its own data.

Worked Example 1

A Safe Bank Account

class BankAccount {
private:
    double balance = 0;

public:
    void deposit(double amount) {
        if (amount > 0) balance += amount;
    }
    void withdraw(double amount) {
        if (amount > 0 && amount <= balance)
            balance -= amount;     // no overdraw
    }
    double getBalance() { return balance; }
};

int main() {
    BankAccount a;
    a.deposit(100);
    a.withdraw(30);
    cout << a.getBalance() << endl;  // 70
}
Expected Output
70

What it does: the balance is private. Deposits must be positive, and withdrawals cannot take more than is there.

Result: the account can never reach an impossible state from outside.

Worked Example 2

A Validated Setting

class Thermostat {
private:
    int temp = 20;     // a safe default

public:
    void setTemp(int t) {
        if (t >= 10 && t <= 30)   // allowed range
            temp = t;
    }
    int getTemp() { return temp; }
};

int main() {
    Thermostat th;
    th.setTemp(25);   cout << th.getTemp() << endl; // 25
    th.setTemp(99);   cout << th.getTemp() << endl; // 25
}
Expected Output
25 25

What it does: the thermostat only accepts a temperature between 10 and 30. The value 99 is rejected, so it stays at 25.

Worked Example 3

A Counter That Cannot Go Wrong

class Counter {
private:
    int count = 0;        // only this class can change it

public:
    void increment() { count++; }
    void reset()     { count = 0; }
    int getCount()   { return count; }
};

int main() {
    Counter c;
    c.increment();
    c.increment();
    cout << c.getCount() << endl;  // 2
    // c.count = 999;  // ERROR: count is private
}
Expected Output
2

What it does: count can only go up by one or reset to zero. Outside code cannot set it to a random value.

Idea: the class controls every way its data can change.

Be Careful

Common Mistakes

  • Making data public
    it can then be changed to anything from outside
  • Forgetting the colon
    write public:, not public
  • Touching private data outside
    obj.balance from main is an error
  • Setter with no check
    a setter that just assigns gives no protection
  • Everything public
    then there is no encapsulation at all
  • Missing the semicolon
    a class ends with };

Rule of thumb: data private, methods public, and validate in setters.

6 · Why private matters

// If balance were public:
//   account.balance = -9999;   // anyone could break it.
// There would be no way to stop bad values.

// Safe version: keep it private and
// change it only through checked methods.
class BankAccount {
private:
    double balance = 0;      // hidden
public:
    void deposit(double a) {
        if (a > 0) balance += a;
    }
    double getBalance() { return balance; }
};

The point: private data plus checked methods means the object stays valid no matter what outside code tries.

Outside code can deposit through a safe path, but it cannot reach in and set the balance to an illegal value.

Wrap Up

Summary

  • Encapsulation bundles data with methods and hides the data inside the object.
  • public members are usable anywhere; private members only inside the class.
  • protected is for inheritance — covered later.
  • The common pattern is private data, public methods.
  • Getters read data and setters change it through a controlled door.
  • Validation in setters keeps an object from ever holding an invalid value.
