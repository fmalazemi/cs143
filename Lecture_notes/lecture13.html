Learning Objectives

  • Tell the difference between a class and an object.

The Story So Far

How We Have Programmed Until Now

So far, our programs kept data and functions apart: variables held the data, and separate functions acted on them.

This works for small programs

  • A few variables and functions are easy to follow.
  • You pass data into functions and get results back.

But large programs get messy

  • Many loose variables are hard to track.
  • It is unclear which function belongs to which data.
  • Changing one part can break another.

OOP offers a tidier way: keep related data and the actions on it together.

The Big Idea

An Object Bundles Data and Behaviour

In OOP, we build programs from objects. An object keeps its own data and the actions that work on that data in one place.

A Familiar Picture

Objects Are All Around Us

Think of everyday things. Each one has properties and can do actions — that is exactly an object.

A car

Has: colour, speed, fuel.

Can: drive, brake, refuel.

A bank account

Has: owner, balance.

Can: deposit, withdraw.

Properties are the data; actions are the behaviour. OOP lets us model things this way in code.

A Key Distinction

Class vs. Object

A class is a blueprint that describes a kind of thing. An object is one real example made from that blueprint.

The Two Parts

Attributes and Methods

Attributes — the data

  • The values an object holds.
  • Describe its current state.
  • Example: a car's colour and speed.

Methods — the behaviour

  • The actions an object can perform.
  • Often read or change its attributes.
  • Example: a car's drive() and brake().

Together they make an object self-contained: it carries its own data and knows how to act on it.

A First Taste

What a Class Looks Like

// a blueprint for a Car
class Car {
public:
    string colour;     // attribute (data)
    int speed;         // attribute (data)

    void drive() {     // method (behaviour)
        speed = speed + 10;
    }
};
  • class Car { ... }; defines the blueprint.
  • Inside, the attributes are the data fields.
  • The method drive() is an action that uses the data.

Do not worry about every keyword yet (like public). We are only seeing the shape today — the rules come next.

A First Taste

Making and Using an Object

int main() {
    Car myCar;          // create an object from the blueprint
    myCar.colour = "red";
    myCar.speed = 0;

    myCar.drive();      // call its method

    cout << myCar.speed; // 10
    return 0;
}
Expected Output
10
  • Car myCar; creates one object.
  • We reach its parts with a dot: myCar.speed.
  • Calling myCar.drive() changes that object's own data.

Each object has its own copy of the data.

Why It Helps

Why Use OOP?

  • Organisation: related data and actions stay together in one place.
  • Modelling: code mirrors real things — a Student, an Account, a Car.
  • Reuse: one class can create many objects, and classes can build on others.
  • Safety: an object can protect its data, allowing changes only through its own methods.
  • Teamwork: large programs split neatly into classes people can work on separately.

In short: OOP helps us manage complexity as programs grow.

The Road Ahead

Four Big Ideas (Coming Later)

OOP rests on four key ideas. We only name them today; each gets its own lecture.

Encapsulation

Keep data and methods together, and protect the data.

Abstraction

Show only what matters; hide the inner details.

Inheritance

Build new classes based on existing ones.

Polymorphism

Let one action behave correctly for different objects.

Concept Answers

Exercise Answers

  • 1 · BankAccount: attributes — owner, balance; methods — deposit(), withdraw().
  • 2 · Class = Car; objects = Sara's Toyota and Ali's Honda (two real cars).
  • 3 · Student: attributes — name, id, gpa; methods — study(), enrol().

There can be more than one good answer — the idea is what matters.

Wrap Up

Summary

  • OOP organises programs around objects instead of loose data and functions.
  • An object bundles its own data (attributes) and behaviour (methods).
  • A class is the blueprint; an object is a real example made from it.
  • OOP helps us model real things and manage complexity as programs grow.
  • The four big ideas — encapsulation, abstraction, inheritance, polymorphism — are coming next.
