The Setup

A Class That Owns Memory

class IntBox {
    int* data;               // a pointer to heap memory
public:
    IntBox(int value) {
        data = new int(value);   // allocate one int
    }
    ~IntBox() {
        delete data;             // free it
    }
    int  get()      { return *data; }
    void set(int v) { *data = v; }
};
  • IntBox holds a pointer to an int on the heap.
  • The constructor uses new; the destructor uses delete.
  • The object does not hold the value directly — it holds the address of the value.

This "owns a pointer" pattern is where copying gets tricky.

First, a Question

When Does a Copy Happen?

IntBox a(5);

IntBox b = a;    // 1. copy when created
IntBox c(a);     // 2. same thing

useBox(a);       // 3. passing by value copies

// 4. returning an object by value copies too
  • A copy is made whenever one object is built from another.
  • It also happens when you pass by value or return by value.
  • So the way copying works matters more often than you might think.

The question: what exactly gets copied when the object owns a pointer?

The Default Behaviour

The Compiler Copies Member by Member

If you do not write a copy constructor, the compiler makes one for you. It copies each member as-is.

For a pointer member, "copy the member" means copy the address — not the thing it points to. Both objects end up pointing to the same memory.

This is called a shallow copy: the pointer is copied, but the memory behind it is shared.

Shallow Copy

Two Objects, One Block

After IntBox b = a; with the default copy, a.data and b.data hold the same address.

Worked Example 1

Problem 1: Shared State

int main() {
    IntBox a(5);
    IntBox b = a;      // shallow copy (default)

    b.set(99);         // change b only...

    cout << a.get() << endl;   // 99  (!)
    cout << b.get() << endl;   // 99
    return 0;
}
Surprising Output
99 99

What went wrong: we changed b, but a changed too. Because both share one block, editing one edits the other.

This "spooky" shared state is the first sign of a shallow copy.

The Bigger Danger

Problem 2: Deleted Twice

At the end of the program both destructors run. Each calls delete data — on the same block.

Freeing memory that is already freed is a serious bug (a "double free").

The Solution

Deep Copy: Give Each Object Its Own Memory

A deep copy allocates new memory for the copy and copies the value into it. The two objects no longer share anything.

Shallow (default)

Copy the pointer → shared block → shared state and double free.

Deep (what we want)

Copy the value into a new block → each object owns its own memory.

We control this by writing a copy constructor.

The Syntax

The Copy Constructor

ClassName(const ClassName& other)
  • Same name as the class — it is a constructor.
  • const ClassName& other — it receives the object being copied, by reference.
  • Inside, you decide how the copy is made — this is where you do the deep copy.

It runs automatically whenever a copy is made.

Deep Copy

Two Objects, Two Blocks

IntBox(const IntBox& other) {
    // new memory, copy the VALUE
    data = new int(*other.data);
}

Each object now points to its own block.

Worked Example 2

Deep Copy in Action

#include <iostream>
using namespace std;

class IntBox {
    int* data;
public:
    IntBox(int value) { data = new int(value); }

    // deep copy constructor
    IntBox(const IntBox& other) {
        data = new int(*other.data);
    }

    ~IntBox() { delete data; }
    int  get()      { return *data; }
    void set(int v) { *data = v; }
};

int main() {
    IntBox a(5);
    IntBox b = a;      // now a DEEP copy

    b.set(99);
    cout << a.get() << endl;   // 5
    cout << b.get() << endl;   // 99
    return 0;
}
Expected Output
5 99

What it does: the copy constructor gives b its own block. Now changing b leaves a untouched, and each destructor frees a different block.

Worked Example 3

Deep Copy of a Dynamic Array

class Numbers {
    int* arr;
    int  size;
public:
    Numbers(int s) {
        size = s;
        arr = new int[size];
        for (int i = 0; i < size; i++) arr[i] = 0;
    }
    // deep copy: new array + copy each element
    Numbers(const Numbers& other) {
        size = other.size;
        arr = new int[size];
        for (int i = 0; i < size; i++)
            arr[i] = other.arr[i];
    }
    ~Numbers() { delete[] arr; }
    void set(int i, int v) { arr[i] = v; }
    int  get(int i)        { return arr[i]; }
};
Using it
Numbers a(3);
a.set(0, 10);
Numbers b = a;     // deep copy
b.set(0, 99);
cout << a.get(0);   // 10
cout << b.get(0);   // 99

Idea: allocate a new array, then copy the elements one by one — never just copy the pointer.

Good Practice

The Rule of Three

If a class needs one of these, it usually needs all three. They keep copying and freeing correct.

  • Destructor
    ~IntBox() — frees the memory
  • Copy constructor
    IntBox(const IntBox&) — deep-copies on creation
  • Copy assignment
    operator= — deep-copies on b = a; after creation
// copy assignment, for reference
IntBox& operator=(const IntBox& other) {
    if (this != &other) {          // avoid self-copy
        delete data;               // free old
        data = new int(*other.data); // deep copy
    }
    return *this;
}
A Short Comparison

How Python Differs

GoalPythonC++
Share the same objectb = ashallow copy
Make an independent copycopy.deepcopy(a)deep copy constructor
Clean up memoryautomaticyou write the destructor

In Python, b = a makes both names refer to the same object — much like a shallow copy. To get an independent copy you call copy.deepcopy(a).

C++ has the same two ideas, but you must write the deep copy yourself — and a wrong copy can crash the program, not just surprise you.

Be Careful

Common Mistakes

  • Trusting the default copy
    when the class owns a pointer, the default is shallow
  • Copying the pointer, not the value
    data = other.data; is still shallow
  • Forgetting new in the copy constructor
    no new block means no real copy
  • Writing only the copy constructor
    assignment b = a; also needs deep copy
  • Missing the self-copy check
    operator= should test this != &other
  • Using delete for an array
    an array needs delete[]

Test to remember: if editing the copy changes the original, it was a shallow copy.

Solutions (1–3)

Exercise Answers

1 · Shallow or deep?

Shallow. The default copy copies the pointer's address, so both objects point to the same memory rather than to separate copies.

2 · Deep copy constructor

IntBox(const IntBox& other) {
    data = new int(*other.data);
}

3 · Why the crash?

Both objects hold the same address, so each destructor calls delete on it — the memory is freed twice, which is a double free and crashes the program.

Exercise Answers

4 · Numbers deep copy

Numbers(const Numbers& other) {
    size = other.size;
    arr = new int[size];
    for (int i = 0; i < size; i++)
        arr[i] = other.arr[i];

b = a shares the list — both names refer to the same list. copy.deepcopy(a) makes a fully independent copy.

Writing data = other.data; is a shallow copy — it copies the address, so both objects share one block.

data = new int(*other.data);
Wrap Up

Summary

  • A shallow copy copies the pointer, so two objects share one block of memory.
  • Sharing causes shared state (editing one edits the other) and a double free at the end.
  • A deep copy allocates new memory and copies the value, giving each object its own block.
  • You control copying with a copy constructor: ClassName(const ClassName& other).
  • The Rule of Three: a class with a destructor usually needs a copy constructor and copy assignment too.
  • Test: if editing the copy changes the original, the copy was shallow.

When an object owns memory, copy it deeply — or not at all.
