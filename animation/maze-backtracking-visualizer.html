      The backtracking recipe
      

        
Three lines, every time

        
Choose, explore, un-choose. Mark the cell, recurse, unmark the cell. Every backtracking problem you will ever meet — N-queens, sudoku, permutations, subsets, graph colouring — is those same three lines with a different notion of “choice” and a different test for “valid”.

        
The undo is the whole idea

        
Without the unmark, the state you leave behind belongs to a branch that failed, and it poisons every branch tried afterwards. The recursion has to hand back the world exactly as it received it. Watch the orange steps: that is the algorithm cleaning up after itself.

        
Pruning is where the speed is

        
The three rejections here — off the grid, a wall, already on the path — kill a branch before a single recursive call is made. Every rejected move is an entire sub-tree of futures never explored. Better pruning beats a faster computer every time.

        
It is depth-first, and it shows

        
Backtracking commits hard to one guess and only reconsiders when reality says no. That is why the first path found is almost never the shortest — it is just the first one the try-order happened to walk into. If you need the shortest path, you want breadth-first search, not this.

        
Try-order changes everything (and nothing)

        
This page tries Down, Right, Up, Left. Change that order and you get a different path, different dead ends, a different number of steps — but the same answer to “is there a path?”. The order is a heuristic, not part of the correctness.

      

    


    

      Things worth trying
      

        
The point of a “perfect” maze

        
Generate a perfect maze and run “Find every path”. The answer comes back as exactly 1 — that is the definition of perfect: between any two cells there is one route and no loops. The same maze under “Find one path” finds that same route.

        
Watch an exponential blow up

        
Set the maze to “Open room” and run “Find every path” on 5×5. There is no cleverness left to save you: the number of routes is the answer, and the answer is enormous. The run will hit this page's step limit and say so. That is not a bug in the code — it is the shape of the problem.

        
Proving a negative

        
Pick “Random walls” until you get an unsolvable maze, or just wall off the exit yourself by clicking. The search visits everything it can reach and returns false. “No path exists” is a real answer, and exhaustive search is how you earn it.

        
The comb: all pruning, no backtracking

        
The comb is one long snake with no branches at all, so its dead-end counter finishes on zero — there is never a choice to regret. Yet a 9×9 comb still rejects about 40 moves across 89 calls. That is the honest lesson: most of what a backtracking search does is not backtracking, it is testing moves and throwing them away. Compare it with a perfect maze of the same size, which hits roughly 13 dead ends.

        
Draw your own trap

        
Click cells to build a corridor that looks promising and dead-ends at the last moment. Watch how much work the algorithm does before it discovers what you already know.
