      Why it works this way
      

    


    

      The classic bugs
      

        
The overflow one

        
Writing mid = (low + high) / 2 looks fine and is wrong in a fixed-width integer language: with a large array, low + high can exceed the maximum int and go negative. Use mid = low + (high − low) / 2, which cannot overflow and computes the same value. This bug sat undetected in the JDK for nine years.

        
The off-by-one one

        
while (low <= high) with high = n − 1. Use < instead and a one-element window is never examined, so the search misses targets sitting at the edges. The two boundary choices must match: inclusive high = n − 1 goes with <=.

        
The infinite one

        
Every branch must shrink the window. Writing low = mid instead of low = mid + 1 means a two-element window where mid lands on low never changes, and the loop spins forever. Excluding mid is safe precisely because the probe already ruled it out.

        
The unsorted one

        
Binary search on unsorted data does not fail loudly — it returns "not found" for values that are sitting right there. The whole method rests on one assumption, and nothing in the code checks it.

      

    


    

      Versus linear search
      

        
The counter tells the story

        
Watch “linear scan would have checked” climb one at a time while binary search does its work in a handful of probes. On 30 values it is 5 probes against up to 30 checks; on a million values it is 20 against a million.

        
But sorting is not free

        
Sorting costs O(n log n), so a single search on unsorted data is faster with a plain linear scan. Binary search pays off when the same array is searched many times — sort once, then every later lookup is nearly instant.

        
Why it needs random access

        
Jumping straight to the middle has to be cheap. On an array, A[mid] is one step. On a linked list, reaching the middle means walking there, so the halving saves nothing and the search degrades to linear.
