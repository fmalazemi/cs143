      Why it works this way
      

        
Divide and conquer

        
A hard problem (sort n values) is cut into two easier copies of the same problem (sort n/2 values), then the answers are stitched together. Cutting costs nothing; all the real work is in the stitching.

        
Why recursion

        
Each half is sorted by exactly the same procedure. Recursion lets you write that once. The halving stops at size 1, which is already sorted — that base case is what keeps it from running forever.

        
Why merging is needed

        
Two sorted halves are not a sorted array. Merging walks both halves at once and always takes the smaller front value, so the result comes out in order in a single pass of n steps.

        
Why it is stable

        
On a tie the algorithm takes from the left half first (left[i] <= right[j]). Equal values therefore keep their original relative order. Change that to < and stability is gone.

        
Why extra memory

        
You cannot merge into the same space you are reading from without overwriting values you still need. Merge copies each half out first, so it uses O(n) additional room — the price for guaranteed n log n behaviour.
