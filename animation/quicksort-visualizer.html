      The pivot problem
      

        
Try this

        
Set the shape to “Already sorted”, the pivot to “Last element”, and run it. The recursion tree comes out as a single stick: every partition peels off one value and hands back a sub-array of n−1. That is n levels deep and n²/2 comparisons — quicksort at its absolute worst, on the friendliest data imaginable. Now switch the pivot to “Median of three” and run the same array again.

        
Why the textbook default is a trap

        
“Last element” is only bad when the data has a pattern — but real data is full of patterns. Sorted, reverse sorted, or nearly sorted inputs are the common case in practice, not the exception, and those are exactly the inputs that make a fixed pivot degenerate.

        
What a random pivot buys

        
The worst case does not disappear — it becomes unreachable on purpose. No input can be constructed to defeat a random pivot, because the algorithm's behaviour no longer depends only on the input. Expected time is O(n log n) for every input, and the odds of a genuinely bad run are astronomically small.

        
Median of three

        
Look at the first, middle and last values and take the middle-sized one. It costs two comparisons and makes sorted and reverse-sorted input into best cases instead of worst cases. This is what most real implementations do, often alongside a random element.

        
The all-identical trap

        
Set the shape to “All identical”. Lomuto's <= test sends every value into the left side, so every partition splits n into n−1 and 0 — quadratic, on an array that needs no sorting at all. Run the same array under Hoare: it splits down the middle instead. Real sorts add a third bucket for values equal to the pivot.

      

    


    

      Quicksort vs merge sort
      

        
Where the work happens

        
Merge sort splits carelessly and works hard to merge. Quicksort works hard to split and then merges for free — once partition is done, the two sides are already positioned correctly relative to each other. It is the same divide-and-conquer skeleton, with the effort moved from after the recursion to before it.

        
Why quicksort usually wins anyway

        
It sorts in place with no auxiliary array, and it sweeps memory in straight lines, which caches love. Merge sort's O(n log n) is guaranteed, quicksort's is only typical — and typical wins on a real machine.

        
Why merge sort still exists

        
It is stable and it never degrades. When a worst case is unacceptable, or ties must keep their order, the guarantee is worth the memory. Many standard libraries hedge: Java uses quicksort for numbers and merge sort for objects, precisely over stability.

        
The hybrid everyone actually ships

        
Introsort: start with quicksort, switch to insertion sort for tiny sub-arrays, and if the recursion gets suspiciously deep, bail out to heapsort. Quicksort's speed, with merge sort's guarantee bolted on.
